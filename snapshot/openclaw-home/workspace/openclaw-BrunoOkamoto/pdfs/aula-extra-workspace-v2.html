Aula Extra — Workspace

Como organizar seu workspace
para escalar

Estrutura de pastas, regras de onde salvar cada coisa, e como o workspace evolui junto com seu agente.

⏱ ~20 min
📂 Prático
🏗️ Fundação para escala
01 — O problema

Workspace bagunçado = agente perdido

Quando você começa com OpenClaw, tudo parece simples. Um arquivo aqui, um script ali. Mas depois de alguns meses, sem estrutura, o workspace vira um caos — e o agente junto.

❌ Workspace bagunçado
Arquivos soltos na raiz
Agente não sabe onde salvar
Reports misturados com rascunhos
Skills sem categoria
Impossível auditar o que está onde
Context window polui com lixo
✅ Workspace organizado
Cada tipo de arquivo tem um destino
Agente sabe exatamente onde salvar
Reports são facilmente consultáveis
Skills categorizadas e documentadas
memory_search encontra qualquer coisa
Contexto limpo e relevante
💡

A regra de ouro

O workspace não é só onde você salva arquivos — é a memória externalizada do seu agente. Quanto mais organizado, mais inteligente ele parece.

02 — Estrutura base

A estrutura que escala

Essa é a estrutura que funciona para qualquer tipo de uso — solo founder, equipe pequena, criador de conteúdo. O exemplo abaixo é o workspace real da Amora — agente da Pixel Educação, usada como Chief of Staff pelo Bruno Okamoto.

🐾

⚠️

Nunca solte arquivos na raiz

A raiz é reservada para os 7 arquivos de identidade. Qualquer output vai numa subpasta — esse é o padrão que faz a Amora saber onde está tudo sem precisar perguntar.

03 — Os 7 arquivos raiz

A identidade do seu agente

Esses arquivos ficam na raiz e são carregados em toda sessão. São a base de tudo.

Arquivo O que vai aqui Atualiza quando?
SOUL.md Personalidade, tom, valores, anti-patterns do agente Quando o agente precisa de ajuste de comportamento
IDENTITY.md Nome, versão, modelo, canais de comunicação, IDs Quando muda canal ou adiciona integração
USER.md Quem você é — negócios, rotina, tom de voz, família Quando muda algo relevante na sua vida/negócio
MEMORY.md Índice de memória — aponta para arquivos em memory/ Quando adiciona novo contexto permanente
AGENTS.md Regras operacionais, startup sequence, red lines Quando muda o comportamento esperado do agente
TOOLS.md Mapa de integrações, credenciais, estrutura de pastas Quando adiciona nova ferramenta ou integração
HEARTBEAT.md Checklist do que checar periodicamente (email, calendário) Quando muda sua rotina de monitoramento
💜

SOUL.md e USER.md são os mais importantes

Um agente sem SOUL.md é um chatbot genérico. Um agente sem USER.md não te conhece. Invista tempo nos dois — é o que faz a diferença entre um agente mediocre e um que parece ter trabalhado com você há anos.

04 — A regra de onde salvar

Cada arquivo tem um destino certo

A maior fonte de confusão em workspaces bagunçados: ninguém (humano nem agente) sabe onde salvar cada coisa. A regra abaixo resolve isso.

Tipo de arquivo Destino
Report / análise (MD, HTML) reports/{categoria}/
Dashboard HTML gerado reports/{categoria}/
PDF entregável reports/{categoria}/
Imagem entregável (PNG, JPG) reports/{categoria}/
Screenshot / imagem temporária archive/imgs/
PRD de projeto novo projects/{nome}/PRD.md
Skill nova skills/{categoria}/SKILL.md
Script operacional scripts/{categoria}/nome.sh
Doc antigo / obsoleto archive/
Memória de projeto ativo memory/projects/{nome}.md
Decisão permanente memory/context/decisions.md
Diário da sessão de hoje memory/sessions/YYYY-MM-DD.md
🧠

A regra simples

Se vai ser consultado de novo → reports/. Se é rascunho ou temporário → archive/. Nunca na raiz.

05 — Memória estruturada

A pasta memory/ como segundo cérebro

A pasta memory/ é onde o agente persiste conhecimento entre sessões. Cada subpasta tem um propósito específico.

📋 context/

Decisões permanentes, lições aprendidas, pessoas relevantes, contexto de negócio. Raramente muda — é a "constituição" do seu agente.

decisions.md lessons.md people.md

🚀 projects/

Um arquivo por projeto ativo. Status, próximos passos, decisões. Quando termina → archive/. Abaixo os projetos reais da Amora:

comunidade-ai.md metricaas.md mgm.md mission-control-v2.md openclaw.md _index.md

🔗 integrations/

Mapa de ferramentas, IDs de canais, credenciais (sempre via 1Password — nunca hardcoded!), configurações de acesso.

credentials-map.md telegram-map.md

📅 sessions/

Diário de sessão: um arquivo por dia (YYYY-MM-DD.md). O agente carrega automaticamente hoje + ontem. O resto é buscado via memory_search quando necessário.

2026-03-14.md 2026-03-13.md
🔍

memory_search encontra qualquer coisa

Não precisa carregar todos os arquivos no contexto. O agente usa memory_search("termo") para buscar semanticamente em toda a pasta memory/ sob demanda — econômico em tokens e muito eficiente.

06 — Skills como capacidades

Organizando suas skills

Governança de skills

Antes de criar uma skill nova, verifique se já existe no ClawHub. E sempre documente: o que a skill faz, quando usar, quando NÃO usar. Skills sem documentação viram lixo em 3 meses.

07 — Evolução contínua

O workspace cresce com você

Um workspace bem estruturado não é estático — ele evolui conforme seu uso muda. Siga esse ciclo.

Comece simples

Não tente criar a estrutura perfeita no dia 1. Comece com os 7 arquivos raiz + memory/ básico. Adicione conforme a necessidade aparecer.

Documente onde salvar no TOOLS.md

Assim que você decide um padrão ("reports de MRR vão em reports/business/"), escreva no TOOLS.md. O agente vai seguir sem precisar perguntar toda vez.

Revise periodicamente

Uma vez por mês: audite o que está em archive/, delete o que não serve mais, e veja se alguma pasta cresceu demais e precisa de subcategorias.

Atualize os arquivos raiz sempre que o workspace crescer

Criou uma nova categoria de skill? Adicionou uma integração nova? Começou um projeto importante? Registre nos arquivos raiz imediatamente. Não adianta ter a estrutura se o agente não sabe que ela existe.

⚠️

O agente só sabe o que você ensinou

Se você criou uma skill de "análise de Meta Ads" mas não atualizou o TOOLS.md e o MEMORY.md, na próxima sessão o agente vai agir como se ela não existisse — e pode criar outra do zero ou deixar de usá-la.

📋 Regra prática: ao criar/adicionar qualquer coisa nova, atualize:
→ Nova skill → TOOLS.md + MEMORY.md
→ Nova integração → TOOLS.md
→ Novo projeto → MEMORY.md (projects/)
→ Nova regra de comportamento → SOUL.md ou AGENTS.md

Deixe o agente sugerir melhorias

Agentes bem configurados (com SOUL.md forte) vão notar quando algo está fora do lugar e sugerir reorganização. Aceite esse feedback — é o sinal de que o sistema está funcionando.

08 — Prompt do aluno

Organize seu workspace agora

Cole esse prompt no seu agente OpenClaw para fazer um diagnóstico e reorganização do seu workspace.

Acabei de assistir a aula sobre organização de workspace. Quero que você faça uma auditoria completa do meu workspace e me ajude a organizar.

**Fase 1 — Diagnóstico:**
1. Liste o que está na raiz do workspace (ls no diretório raiz)
2. Identifique arquivos que estão fora do lugar
3. Verifique se os 7 arquivos raiz existem: SOUL.md, IDENTITY.md, USER.md, MEMORY.md, AGENTS.md, TOOLS.md, HEARTBEAT.md
4. Me diga o que está faltando ou mal posicionado

**Fase 2 — Estrutura:**
5. Crie as pastas que estão faltando:
- memory/context/ (decisions.md, lessons.md, people.md)
- memory/projects/
- memory/integrations/
- memory/sessions/
- skills/ (com subpastas: content/, analytics/, operations/, research/)
- reports/ (com subpastas: business/, misc/)
- archive/

**Fase 3 — Regras no TOOLS.md:**
6. Adicione ao TOOLS.md uma seção "Onde salvar cada output" com a tabela de destinos
7. Defina as categorias de reports que fazem sentido para meu uso

**Fase 4 — Teste:**
8. Me faça criar um output de teste (um pequeno relatório) e salve no lugar certo
9. Confirme que você consegue encontrá-lo depois com memory_search

Objetivo: no final dessa sessão, meu workspace deve ter estrutura clara e você deve saber exatamente onde salvar qualquer tipo de arquivo sem precisar me perguntar.
🎯

Dica: use o workspace da Amora como referência

O workspace mostrado nessa aula é o workspace real da Amora (agente da Pixel Educação). Você pode usá-lo como template — adapte as categorias para o seu contexto.

09 — Gestão de projetos

Onde ficam as tasks e pendências?

Essa é a pergunta que todo mundo faz na hora de organizar o workspace. E a resposta honesta é: não existe receita única. O que existe é um framework de decisão.

🧭

A regra de ouro

Escolha uma fonte de verdade para suas tasks e faça o agente sempre apontar para ela. O problema não é qual ferramenta — é ter duas fontes ao mesmo tempo.

As 3 opções

📝

pending.md por projeto

mais simples

Cada projeto tem um arquivo memory/projects/{projeto}.md com uma seção ## Pending. O agente lê e atualiza diretamente, sem API, sem setup.

✅ Zero dependência externa  ·  ✅ Agente acessa direto  ·  ❌ Sem interface visual

🗂️

Notion / Linear / Trello

mais visual

Tasks vivem na sua ferramenta favorita. O agente acessa via API (Notion tem integração nativa no OpenClaw). Você visualiza no browser com kanban, filtros, datas.

✅ Interface visual  ·  ✅ Colaborativo  ·  ❌ Depende de API  ·  ❌ Mais setup

🚀

Dashboard próprio (Mission Control)

máximo controle

Você constrói seu próprio painel de gestão — conectado ao agente, ao Notion, às métricas do negócio. Controle total, visual customizado, sem depender de terceiros.

✅ Controle total  ·  ✅ Integrado com agente  ·  ❌ Requer build  ·  ❌ Mais complexo

A regra que funciona em todas as opções

Independente de qual ferramenta você escolher, faça isso em cada arquivo de projeto:

# memory/projects/meu-projeto.md

## Tasks
# Opção A: link direto pro Notion
Tasks: https://notion.so/seu-workspace/meu-projeto

# Opção B: pending aqui mesmo
- [ ] Tarefa pendente
- [x] Tarefa concluída
💡

Isso funciona com qualquer ferramenta

Notion, Obsidian, Linear, Trello, ClickUp — não importa. O que importa é que o agente saiba onde buscar quando você mencionar um projeto. Com uma linha de referência no arquivo do projeto, ele sempre sabe.

10 — Checklist

Seu workspace está pronto quando...

  • Os 7 arquivos raiz existem e estão preenchidos (SOUL, IDENTITY, USER, MEMORY, AGENTS, TOOLS, HEARTBEAT)
  • A pasta memory/ tem as subpastas: context/, projects/, integrations/, sessions/
  • Skills estão em skills/{categoria}/SKILL.md
  • TOOLS.md tem a tabela "onde salvar cada output"
  • reports/ tem subpastas por categoria (business/, misc/, etc.)
  • Não há arquivos soltos na raiz além dos 7 arquivos de identidade
  • O agente consegue responder "onde devo salvar um relatório de MRR?" sem precisar perguntar
  • memory_search consegue encontrar algo que você pediu pra salvar ontem
  • Cada projeto em memory/projects/ tem uma referência para onde estão as tasks (Notion, pending.md, ou outra ferramenta)
