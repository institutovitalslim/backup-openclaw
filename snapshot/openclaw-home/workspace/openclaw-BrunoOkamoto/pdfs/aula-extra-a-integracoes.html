03
gog CLI NOVO 3.2

Google Workspace de forma simples — sem criar projeto no Google Cloud

⚠️
Por que não usar o fluxo OAuth manual?

O fluxo OAuth manual para Google exige criar projeto no Google Cloud Console, configurar OAuth credentials, habilitar APIs... É complicado e quebra fácil. O gog CLI elimina tudo isso.

1
Instalar gog CLI
npm install -g gog
2
Autenticar (um comando para tudo)
# Autentica Calendar, Drive, Gmail e Docs de uma vez gog auth login

Uma janela do browser vai abrir. Autorize com sua conta Google. Pronto — todos os serviços ficam autenticados.

3
Usar os serviços
# Google Calendar gog calendar list --account=SEU_EMAIL gog calendar create --title="Reunião" --start="2026-03-10T14:00" # Google Drive gog drive ls --account=SEU_EMAIL gog drive upload /caminho/arquivo.pdf --parent ID_PASTA # Gmail gog gmail inbox --limit 10 --account=SEU_EMAIL gog gmail send --to=fulano@email.com --subject="Oi" # Google Docs (leitura) gog docs get ID_DO_DOC --account=SEU_EMAIL
💡
Dica: use variável de ambiente

Configure GOG_ACCOUNT=seu@email.com no ambiente para não precisar passar --account em todo comando.

05
APIs Diretas

Quando não tem skill pronta — o passo a passo

Qualquer ferramenta que tem uma API pode ser integrada. Siga o fluxo:

1
Conseguir a API Key

Acesse o painel da ferramenta → Developers / API → crie uma chave.

🔴
NUNCA hardcode a API Key

Não coloque a chave direto no código, config.json ou qualquer arquivo. Use o 1Password.

2
Guardar no 1Password
# Via CLI (se tiver o op instalado) op item create --category=api-credential \ --title="Stripe Produção" \ --vault="Amora Vault" \ secret_key="sk_live_..."

Ou adicione manualmente pelo app/web do 1Password.

3
Agente busca a credencial quando precisar
# O agente faz isso nos bastidores: STRIPE_KEY=$(op item get "Stripe Produção" \ --vault "Amora Vault" --field secret_key --reveal) curl https://api.stripe.com/v1/balance \ -u "$STRIPE_KEY:"
4
Documentar no TOOLS.md
### Stripe (Pagamentos) - **Item 1PW:** `Stripe Produção` - **Vault:** Amora Vault - **Acesso:** Read livre, Write = aprovação obrigatória
06
Skill Customizada

Para ferramentas nichadas — uma pasta com instruções

# Criar a pasta da skill mkdir -p ~/.openclaw/skills/crm-empresa-x # Criar o SKILL.md (só texto, sem código!) # O agente lê e segue as instruções # Registrar openclaw skills register ~/.openclaw/skills/crm-empresa-x # Verificar openclaw skills list
💡
Você não precisa programar

O SKILL.md é texto puro. Copie a documentação da API da ferramenta, explique como o agente deve usar, pronto. O agente faz o resto.

07
Fluxo de Decisão

Como integrar qualquer ferramenta — 5 perguntas

🗺️ Árvore de Decisão
  1. A ferramenta tem API? (procure "API documentation" no Google)
  2. Tem skill no ClawHub? → clawhub search nome
  3. Se sim → clawhub install nome → testar
  4. Se não → pegar API key → guardar no 1Password → documentar no TOOLS.md
  5. Se vai usar frequentemente → criar skill customizada

🔒 Regras de Segurança — Invioláveis

  • NUNCA colocar API keys direto no código ou arquivos de config
  • NUNCA commitar credenciais no Git
  • SEMPRE usar 1Password (ou gerenciador equivalente)
  • SEMPRE definir permissões (read-only quando possível)
  • SEMPRE documentar no TOOLS.md o que o agente pode e não pode fazer
# Auditoria de secrets (nunca deixe chaves expostas) openclaw secrets audit # Se encontrar algo → migrar pro cofre seguro openclaw secrets apply

🔧 Troubleshooting Comum

❌ "clawhub: command not found"
npm install -g clawhub
❌ "Skill instalou mas agente não sabe usar"
openclaw skills list → se não aparece: openclaw skills register ~/.openclaw/skills/nome → openclaw gateway restart
❌ "API retorna 401 Unauthorized"
Verificar se a API key está correta, se não expirou e se tem permissões suficientes. Gere uma nova chave se necessário.
❌ "gog: authentication failed"
Rode gog auth login novamente. Se persistir, verifique GOG_KEYRING_PASSWORD configurado no ambiente.
