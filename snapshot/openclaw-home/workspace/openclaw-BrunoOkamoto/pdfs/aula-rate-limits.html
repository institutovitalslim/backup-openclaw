01
O que é Rate Limit?

O provedor limita quantas requisições você pode fazer por unidade de tempo.

Rate limit é um mecanismo de proteção que todo provedor de API usa. Quando você ultrapassa a cota, a API recusa a chamada com o status HTTP 429 — "Too Many Requests". Não é falha sua; é uma regra de infraestrutura que existe por boas razões:

Motivo 1
🏗️ Proteger infraestrutura
  • Evitar sobrecarga dos servidores
  • Garantir estabilidade da plataforma
  • Prevenir abusos e DDoS acidentais
Motivo 2
🤝 Qualidade para todos
  • Nenhum usuário monopoliza recursos
  • SLA justo para toda a base
  • Tiers refletem uso e comprometimento

📊 Os 3 tipos de limite

RPM
Requests Per Minute
Nº de chamadas por minuto — o mais comum
TPM
Tokens Per Minute
Volume de tokens (input + output) por minuto
RPD
Requests Per Day
Limite diário de chamadas (tier Free)
02
Como aparece no OpenClaw

O sinal mais óbvio é o 429 no log — mas nem sempre fica tão claro.

ERROR RateLimitError: 429 Too Many Requests at Anthropic.makeRequest (sdk/client.js:142) WARN Rate limit reached for model claude-sonnet-4-5 x-ratelimit-remaining-requests: 0 x-ratelimit-remaining-tokens: 0 x-ratelimit-reset-requests: 60s INFO Retrying in 32s (attempt 1/3)…

🔍 Quando acontece mais

🕙 Muitos crons no mesmo horário
10 crons agendados para 9:00h disparam ao mesmo tempo → burst de RPM em segundos. O agente "trava" na hora do rush.
🤖 Sub-agents em paralelo
Cada sub-agent faz suas próprias chamadas. Três sub-agents rodando juntos = 3× a pressão no rate limit da mesma API Key.
💓 Heartbeats muito frequentes
Heartbeat configurado a cada 5 min com modelo pesado + contexto grande → acumula TPM rapidamente ao longo do dia.

04
Estratégias no OpenClaw

Quatro abordagens complementares — use a combinação certa pro seu nível de uso.

🔻 A) Reduzir Pressão

Antes de tentar qualquer outra coisa: gaste menos cota. Parece óbvio, mas a maioria dos problemas se resolve aqui.

⚡ Modelo mínimo para cada tarefa
Haiku para heartbeats e crons rotineiros — usa 10× menos tokens que Sonnet.
Sonnet para interação diária e tarefas com contexto médio.
Opus somente quando precisar de raciocínio profundo — nunca como padrão.
🕐 Espaçar os crons
Nunca agendar múltiplos crons para o mesmo horário exato. Distribua com pelo menos 5 minutos de intervalo:
9:00 → resumo diário  |  9:05 → checar emails  |  9:10 → sync calendário
🤖 Evitar sub-agents em paralelo
Se uma tarefa pode ser feita em sequência, não dispare em paralelo. Cada sub-agent consome cota da mesma API Key. Prefira serial quando não há ganho real de velocidade.

🔄 B) Retry com Backoff Automático

O OpenClaw já cuida disso por você. Quando recebe um 429, o runtime espera e tenta novamente automaticamente.

1
API retorna 429
Limite atingido — chamada rejeitada pelo provedor
2
OpenClaw inicia backoff
Espera exponencial: 1s → 2s → 4s → 8s… até a janela resetar
3
Retry automático — sem intervenção
A chamada é retentada. Você não precisa fazer nada.
💡
Agente "lento" não é agente "travado"

Se você perceber que o agente demorou 30-60 segundos a mais para responder, provavelmente estava em backoff. Isso é comportamento correto — não é falha. O agente vai responder; só precisa aguardar.

📅 C) Distribuir Carga

Quando reduzir e esperar não bastam, a próxima fronteira é distribuir o trabalho no tempo e no espaço.

# ❌ Evite: 3 crons no mesmo minuto crons: - schedule: "0 9 * * *" name: resumo-diario - schedule: "0 9 * * *" name: checar-emails - schedule: "0 9 * * *" name: sync-calendario # ✅ Prefira: distribuídos em 5 min crons: - schedule: "0 9 * * *" name: resumo-diario - schedule: "5 9 * * *" # 09:05 name: checar-emails - schedule: "10 9 * * *" # 09:10 name: sync-calendario
🔒 Jobs pesados → sessionTarget: isolated
Use sessionTarget: "isolated" para sub-agents que fazem muitas chamadas. Isso garante que o job pesado não compete com as interações da sessão principal pelo mesmo rate limit window.

06
Rate Limit e Custo

Esclarece uma confusão frequente — são duas coisas diferentes.

🚦
Rate Limit
Limita a velocidade das chamadas. Quando atingido, o agente espera. Não aumenta o custo — tokens de espera não existem. Apenas atrasa a execução.
💳
Billing Limit
Limita o gasto mensal. Configurável no dashboard. Quando atingido, as chamadas são bloqueadas até o ciclo resetar ou o limite ser aumentado.
429
Rate Limit — espere
402
Billing Limit — pague
$0
Custo do tempo esperando
💡
Rate limit não te cobra nada a mais

Você paga apenas pelos tokens que o modelo processou. O tempo em backoff não gera cobrança. Um agente que ficou 2 minutos em retry usou exatamente os mesmos tokens de um que respondeu em 10 segundos.

07
Dica Avançada: Múltiplas API Keys

Para uso intenso, distribuir entre chaves (ou provedores) é a saída.

🔑 Múltiplas chaves no mesmo provedor
Cada API Key tem seu próprio rate limit independente. Criar duas keys na Anthropic e rotacionar entre elas dobra o seu RPM efetivo. Funciona bem para workloads previsíveis.
🌐 Provedores diferentes por agente
Anthropic para agentes de produção críticos.
OpenAI como backup ou para tarefas específicas de código.
O OpenClaw suporta múltiplos providers nativamente — configure tokens distintos por agente.
# OpenClaw — configurar providers por agente # Agente principal usa Anthropic openclaw config set model anthropic/claude-sonnet-4-5 # Agente de backup configurado com OpenAI # (via OPENAI_API_KEY no env ou config) openclaw config set model openai/gpt-4o --agent backup-agent
🧠
Regra prática

Para 90% dos usuários do curso, o rate limit some depois de gastar $5 (Tier 1) e espaçar os crons. Múltiplas keys e multi-provider são para quem está escalando sério — não pule para isso primeiro.

✅ Checklist Anti-Rate-Limit
Heartbeat usando claude-haiku-4-5 (não Sonnet ou Opus)
Crons distribuídos com ≥ 5 min de intervalo entre si
Jobs pesados com sessionTarget: "isolated"
Tier verificado em console.anthropic.com → Settings → Limits
Billing limit configurado (não deixar ilimitado)
